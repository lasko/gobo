# Gobo

A **privacy-first Android client for [online-go.com](https://online-go.com) (OGS)** — play Go without trackers. Jetpack Compose UI, OAuth2 (PKCE) login, REST + WebSocket against the OGS public API.

The privacy stance is the product, not a nice-to-have. Keep it intact (see [Non-negotiable constraints](#non-negotiable-constraints)).

## Tech stack

- **Language:** Kotlin 2.0.21, JVM target 17
- **UI:** Jetpack Compose (BOM `2024.09.00`, Material3), single-Activity
- **Async:** Coroutines + `StateFlow`
- **Net:** OkHttp 4.12 (REST + WebSocket), kotlinx.serialization JSON 1.7.3
- **Auth:** AppAuth 0.11.1 (OAuth2 + PKCE via Custom Tab), `androidx.security.crypto` EncryptedSharedPreferences for token at rest
- **Build:** AGP 8.13.2, Gradle 8.14.4 wrapper. `minSdk 26`, `targetSdk 34`, `compileSdk 36`

## Design system

Flat, geometry-first, Material3. **No hardcoded hex in components** — read `MaterialTheme.colorScheme` or `LocalBoardColors`.
- **Color:** hand-tuned light/dark schemes in `ui/Theme.kt` (no Material You dynamic color). Accent is mint `#20C997` (light) / cyan `#06B6D4` (dark). The goban's own palette (background, grid, stone fills/borders, last-move) lives in `board/BoardColors` and is provided through `LocalBoardColors` per theme — the board surface matches the app background (no wood skin).
- **Type:** Poppins, **bundled locally** in `res/font` (OFL, license in `assets/`). Do **not** switch to the `GoogleFont`/downloadable-fonts API — it fetches from Google's servers (violates the privacy charter).
- **Board:** stones are 45% of cell; white gets a border only in light, black only in dark; last move marked with the accent. `GoBoard` also renders a translucent `ghostMove` (two-tap preview), a red `invalidCell` flash, and green `hintCells` blocks (the puzzle hint).

### Releases / CI (#46)

`.github/workflows/release.yml` builds a **versioned release APK**. Push a `v*` tag → it runs `testDebugUnitTest` + `lint`, builds `assembleRelease`, asserts the APK is **`INTERNET`-only** (`aapt dump permissions` — the app's own `com.gobo.app.*` signature permission is allowed; any other `android.permission.*` fails the build), prints the signing cert, and publishes a **GitHub Release** with the APK attached. `workflow_dispatch` does the same minus the Release (artifact only).

- **Versioning follows [SemVer](https://semver.org)** and is **tag-driven**. To release: bump `versionName` per SemVer (MAJOR = breaking, MINOR = feature, PATCH = fix — we're pre-1.0, so churn lives in MINOR/PATCH), then `git tag vX.Y.Z` on `main` and push the tag. CI passes the tag (minus `v`) as `-PversionName`; **`versionCode` is *derived* from it** in `build.gradle.kts` via `semverToVersionCode` (`MAJOR*1_000_000 + MINOR*1_000 + PATCH`, each component < 1000 → e.g. `0.2.0`→`2000`, `1.4.3`→`1_004_003`), so it increases in lockstep and there's one number to maintain. The `versionName` literal in `build.gradle.kts` is the local-build / next-release default — **bump it to match each release** in the same change.
- **Signing** is keystore-from-Secrets when present, **debug-key fallback otherwise** — so a secret-less build still produces an installable (testing-only) APK. Real signing is configured: repo Secrets `RELEASE_KEYSTORE_BASE64` (base64 of the `.jks`), `RELEASE_KEYSTORE_PASSWORD`, `RELEASE_KEY_ALIAS`, `RELEASE_KEY_PASSWORD`. The keystore lives **only** in Secrets, never the repo. The signing path keys off the `KEYSTORE_FILE` env var the workflow sets after decoding.
- **R8/minify:** the release build minifies; `proguard-rules.pro` has a `-dontwarn com.google.errorprone.annotations.**` because `security-crypto`→Tink references compile-only Error Prone annotations (a release-only failure the debug build hides).
- Build it locally with `.\gradlew.bat assembleRelease` (uses the literal version) or override: `.\gradlew.bat assembleRelease "-PversionName=0.3.0"` (quote the `-P` arg in PowerShell; `versionCode` is computed from it).

## Architecture notes

- **Single Activity, Compose-only.** `MainActivity` holds a `Session` sealed interface (`NotLoggedIn` → `LoadingConfig` → `Ready`). When `Ready`, `ReadyApp` renders a `ModalNavigationDrawer` (My Games / New Game / Log out). An open game takes over the full screen with its own back nav.
- **ViewModel + StateFlow.** Each screen has a ViewModel exposing a private `MutableStateFlow` and a public `asStateFlow()`. UI state is a sealed interface (`Loading` / `Ready(data)` / `Error(msg)`), plus a `Submitting`/`Success`/`Idle` pattern for actions. Match this when adding screens.
- **Server is authoritative.** `BoardState` applies the OGS `board[][]` snapshot directly; it also has local flood-fill capture logic so incremental `game/move` events render captures without a round-trip. Don't reimplement game rules client-side beyond this responsiveness shim.
- **REST vs realtime split.** `OgsRest` = request/response (config, overview, challenges). `OgsSocket` = realtime game play + broadcasts. The socket speaks raw JSON arrays: client `["command", data, id?]`, server `["event_name", data]`.
- **Finished-game review.** When a game ends with a board, `GameViewModel` initializes a `ReviewState` at the final position; the finished-game screen steps through it move-by-move (`reviewStep`/`reviewJump` → ⏮ ◀ ▶ ⏭, with a "move N / total" counter). Each position is a pure `replayMoves` of the move-list prefix (same logic as the live board), so the board, last-move marker, and prisoner counts are all rebuilt — no extra server round-trip. Back is the top-bar "← Games". (Even games only — replay starts from an empty board, so handicap stones placed outside the move list wouldn't show.)
- **Spectating (#7).** The "Watch Game" drawer destination is a **browse list of live games** (`LiveGamesViewModel` opens its own socket, authenticates, and runs `gamelist/query` — built/parsed by `buildGameListQuery`/`parseGameList` in `net/LiveGames.kt`); tap a game to watch it. It also has a "watch by link" field for a shared game number/OGS URL (`parseGameId` in `net/GameRef.kt`). Either way it opens the normal `GameScreen` in **read-only** mode (`spectating = true`): an observer `game/connect` (no `player_id` thanks to #26, `chat:false`). The watcher is neither player, so `myColor` is null → no board taps, no action row; the status shows whose move (`GameViewModel.toMove`). Reuses everything else — live moves, the reconnect banner, and finished-game review (watching a *finished* game id just opens it in review). `gamelist/query` is a **request/response** call: `OgsSocket.request` sends it with a request id and suspends for the matching numeric-id reply (a `CompletableDeferred` per id; replies are the `[id, data]` frames `onMessage` now routes). The list socket is **scoped to the screen** via a `DisposableEffect` that calls `close()` — the screen creates VMs with `remember`, not `viewModel()`, so `onCleared` won't fire on its own (the game screen got the same `DisposableEffect` cleanup, important now that a stray socket auto-reconnects).
- **My Games live-refresh.** The list has no socket subscription; instead `GameListScreen` polls `GameListViewModel.refresh()` every ~10s **only while RESUMED** (`repeatOnLifecycle`), so turn badges and accepted challenges update without the manual Refresh button and polling stops when backgrounded. `refresh()` is silent (swaps in fresh data on success, keeps the current list on a blip — no Loading flash); `load()` is the spinner-showing variant for the initial load and explicit actions. Just REST GETs — no analytics, no new permissions.
- **GoTV (#42).** The "GoTV" drawer destination lists OGS' **external** live Go streams (Twitch/YouTube) via `GET /api/v1/gotv/streams` → `parseGoStreams` → `GoTvViewModel`. **These are not OGS games** (no game id, nothing to spectate in-app), so it does *not* reuse the spectate path — tapping a stream **opens it in the device browser** (`LocalUriHandler.openUri`, URL from `streamUrl`), the same privacy hand-off as the OAuth Custom Tab: we never embed a third-party player (that would pull in a tracker-laden WebView) and add **no new permission**. We render OGS' own JSON text only and **deliberately ignore** the payload's `thumbnail_url`/`profile_image_url` — they're on Twitch's CDN, so loading them would leak each user's IP/requests to a third party. Streams with an unknown `source` (no buildable URL) are dropped. REST GET only, no analytics.
- **Puzzles (#40).** The "Puzzles" drawer destination browses public tsumego **collections** (`PuzzlesViewModel` → `parsePuzzleCollections`); tap one to **solve its starting puzzle on the board**. The list mirrors the OGS web puzzle browser: each row shows difficulty (`formatDifficulty`), puzzle count, rating, views, solved count, owner, and created date (`formatPuzzleDate`/`formatCount`), and a chip bar **sorts server-side** (`PuzzleSort` → OGS' `ordering` param; re-tapping the active field flips direction) so the top of a 4000+ list is the good stuff. Default sort is best-rated. Solving takes over the screen like an open game (`puzzleId` short-circuit in `ReadyApp`, its own back nav), driven by `PuzzleViewModel`. A **Hint** button (💡, toggle) highlights the correct move(s) for the current position — `puzzleHints` is a **minimax** over the tree (the branches from which the player can force a solution), drawn as a translucent green block via `GoBoard`'s `hintCells` + `BoardColors.hint`; it clears once you play. (A shallow "non-wrong branch" filter is wrong because OGS authors usually flag `wrong_answer`/`correct_answer` on the *opponent's reply* a level deeper than the player's move — see the cheat-sheet.) **No socket — pure REST GETs**, no analytics; we never post solve results. All logic is in `net/Puzzles.kt` (pure, unit-tested): models, the `parse*`/`format*` functions, and `puzzleStep` (the solution-tree walk). The board reuses `GoBoard` for input and `BoardState.place` for captures (life-and-death puzzles capture stones), and the invalid-cell flash for off-tree taps. The solve screen pins the action row + tap hint to the bottom and puts the status/description/board in a scrolling region above, with the (often long, teaching) description collapsed to a few lines + a Show more/less toggle — so a wordy puzzle can't push the controls off-screen. The non-obvious OGS bits live in the cheat-sheet below. (Server-side *search* isn't wired — the collections endpoint ignores a `search=` param; sorting covers discovery for now.)

## OGS integration cheat-sheet

- **All endpoints live in `net/Ogs.kt`** — add new ones there, never inline a URL, so the network surface stays auditable.
- **Coordinates:** OGS moves are two letters, column-then-row `a`..`s` (0-indexed); `".."` = pass. Encode/decode via `board/OgsCoord`.
- **Online bots:** the socket broadcasts an `active-bots` event on connect (no auth needed) — a map keyed by bot id with `username`, `ranking`, and a `config` block (`allowed_board_sizes`, `allowed_time_control_systems`, `allow_ranked`, `decline_new_challenges`, …). Parsed by `parseActiveBots` in `net/Challenges.kt`. This is how the New Game screen lists only live, accepting bots.
- **Challenges:** `POST /api/v1/challenges/` = open seek (any human); `POST /api/v1/players/{id}/challenge/` = direct challenge. `OgsRest.buildChallengeBody` builds the body — it must **mirror the OGS web client's** (`online-go.com` ChallengeModal) exactly, because the **open-seek endpoint 500s on a partial body** (the bot endpoint is forgiving, which masked this — bots worked, human seeks didn't, until #21). The non-obvious bits: top-level `invite_only:false` and `rengo_auto_start:0` are **required**; `min/max_ranking: -1000/1000` is OGS' "no restriction" sentinel (it does **not** clamp — it's just accepted); `komi` is **omitted** in automatic mode (the server computes it); and `time_control_parameters` duplicates the system under a `time_control` key. The POST returns `{"status":"ok","challenge":<id>,"game":<id>}` **optimistically** — a `game` id even when the opponent hasn't accepted yet.
- **Open challenge = waiting for an opponent.** OGS expires an open seek within seconds unless the creator keeps it alive, so an open seek **can't** be posted-and-left in a Pending list. Instead, creating one opens the game screen straight into a **waiting-for-opponent** state: `GameViewModel.start(gameId, chatEnabled, challengeId)` connects (the optimistic POST returns a placeholder `game` id) and, while `GamePhase.Connecting`, pings `challenge/keepalive {challenge_id, game_id}` ~1/s (`challengeKeepAlive`) instead of running the 25s bot timeout. When an opponent accepts, **their acceptance is the `gamedata` snapshot** that arrives on the connect → we flip to `Playing` and the keepalive loop exits. The Connecting screen shows "Waiting for an opponent…" + a **Cancel** (`cancelOpenChallenge` → `DELETE /api/v1/me/challenges/{id}`, then navigate away, which closes the socket so the seek lapses). Bot challenges skip all this (they start immediately). (The `/me/challenges` list + `parseSentChallenges` + the My Games "Pending" section remain for any *direct* challenges but are unused for open seeks now.)
- **Bot challenges can still be declined** after that "ok": the bot sends a `notification` socket event with `type:"gameOfferRejected"` (with `game_id`, a human-readable `message`, and `rejection_details.rejection_code`, e.g. `too_many_games` — most `*-humanlike-kata` bots allow only **1** concurrent game). `GameViewModel` listens for this so a declined challenge surfaces the reason instead of stranding the user on a never-loading board.
- **Playing a game (realtime):** send `game/connect` with `{game_id, chat}` **after `authenticate`** — the JWT identifies the player, so **no `player_id` is needed** (we used to send one; removed in #26 after confirming live that gamedata still arrives without it — goban omits it too). The `game/<id>/gamedata` snapshot (board, players, clock, `phase`) is the signal the game truly exists; `GameViewModel` stays in a "connecting" state until it arrives. (Order matters: authenticate first, then connect — an unauthenticated connect isn't identified and won't get the snapshot, which is what the old `player_id` was really working around.) A finished game re-sends `gamedata` with `phase:"finished"` plus `outcome`/`winner`; resign/timeout may instead come as a `game/<id>/phase` event (`GameViewModel` handles both, falling back to `GET /api/v1/games/{id}` for the result). `GamePhase` (in `GameViewModel`) models the screen lifecycle: `Connecting → Playing → Scoring → Over`.
- **Disconnect/reconnect:** `OgsSocket` owns recovery. `WebSocketListener.onFailure`/`onClosed` (anything but our own code 1000) triggers a reconnect with a short jittered backoff ladder (`reconnectDelayMs` in `net/Reconnect.kt` — fast first tries, capped at a few seconds for battery). It stores the `onOpen` callback and **re-runs it on every reconnection**, so the re-sent `authenticate` + `game/connect` make the server push a fresh `gamedata` snapshot that rebuilds the whole position — **no separate catch-up protocol** (our handlers are idempotent: `handleSnapshot` rebuilds board/moves/captures/clock, `appendChat` de-dupes). `close()` sets a `manuallyClosed` guard so a deliberate teardown isn't treated as a drop. The lifecycle is exposed as a `ConnectionState` flow → `GameViewModel.reconnecting` → a "Reconnecting…" banner (the board freezes but reads as reconnecting, not hung). Privacy: reconnection adds **no** analytics/telemetry messages (no `net/connects`/`net/timeout`); the keepalive ping loop keeps running and resumes naturally once the new socket is up.
- **In-game chat (opt-in):** off by default to keep the connect minimal — `game/connect` sends `chat:false` and no chat flows. The `SettingsStore.chatEnabled` toggle flips `game/connect` to `chat:true`, after which the server pushes `game/<id>/chat` events shaped `{channel, line:{username, player_id, date, chat_id, body}}` (older payloads wrap the line under `message`; `parseGameChat` accepts either). The `line.body` is one of three shapes: a **plain string** (human chat); a **`{"type":"translated","en":…,<lang>:…}` object** — bots send their greeting and end-of-game notes this way, and we render the `en` text; or an **`analysis`/`review` variation object**, skipped because the read-only view shows text only. `chatBodyText` in `net/GameChat.kt` does this discrimination (**don't** treat every non-string body as skippable — that drops all bot chat, the bug fixed here). `GameViewModel` collects the lines into `chat`. **OGS re-sends the whole chat log on some lifecycle events (notably at game end, prefixed by a `game/<id>/reset-chats` event)**, so each line carries a `chat_id` and `appendChat` de-dupes on it — chat appends must stay idempotent or the history doubles. A bot's opening greeting is **not** in the gamedata snapshot (these games carry no `chat_log` field); it arrives as a live `game/<id>/chat` event once the bot acts and is replayed at game end, so de-dup is what keeps it single. `parseGameChatLog` still recovers a snapshot `chat_log` when one is present (also funneled through `appendChat`). Because of replay=0, `GameViewModel` subscribes to events **before** calling `game/connect` so the post-connect burst isn't missed.
- **Sending chat:** `game/chat {game_id, body, move_number, type}` — the *current* protocol authenticates via the earlier `authenticate` message, so **no per-message auth/identity is needed** (the legacy `game_chat_auth`/`username`/`ranking`/`ui_class` fields the readme docs show are gone — confirmed against the goban `ClientToServer` protocol types). `type` is `"main"` for normal play. Built by `buildGameChatMessage`, sent via `OgsSocket.sendChat`; `GameViewModel` tracks `move_number` and relies on the server echoing the line back (as a `game/<id>/chat` event) to render your own message — no optimistic insert.
- **Clocks:** OGS pushes a `game/<id>/clock` event on every move (also embedded in the gamedata snapshot as `clock`) — **not** every second, so a live countdown ticks locally off the anchor. Shape: `{current_player, black_player_id, white_player_id, last_move (epoch-ms anchor), expiration, black_time, white_time, now?}`. Each `*_time` entry varies by system: byoyomi `{thinking_time, periods, period_time, period_time_left?}`, fischer `{thinking_time, skip_bonus}` (the increment is already baked into `thinking_time` server-side — never add it client-side), absolute `{thinking_time}`, simple a **bare number** = the move deadline (unverified live — handled defensively). Only the `current_player`'s side counts down; the other is frozen. Parsed by `parseClock` in `net/GameClock.kt`; `readClock` computes the live remaining time (main time → byoyomi periods) at a given instant and `formatClock` renders it (`M:SS`/`H:MM:SS`, `+(n)` periods in byoyomi). `GameViewModel` exposes a `clock` StateFlow; `GameScreen` runs a 1 Hz tick (only while `Playing`) and uses the optional `now` field to derive a device→server skew so the countdown tracks server time. Shown in the player tags with `FontFamily.Monospace` + tabular figures so the digits don't jitter (privacy-clean system font — no bundled binary, no downloadable-fonts API).
- **Undo:** request a take-back with `game/undo/request {game_id, move_number}` (`move_number` = the current move count — matches goban's `requestUndo`); built by `buildUndoRequest` in `net/Undo.kt`, sent via `OgsSocket.requestUndo`. **Nothing reverts optimistically** — the opponent must accept. On request the server echoes a `game/<id>/undo_requested` (`{move_number, requested_by, undo_move_count}`, `requested_by` = us) to confirm receipt; the actual revert only comes if the opponent sends `game/<id>/undo_accepted` (or `undo_canceled` if withdrawn). **Many bots ignore undo — amybot-beginner never accepts (verified live; OGS's own web client behaves the same), so it's a no-op there.** Because acceptance is uncertain, `GameViewModel` exposes `undoRequested` and the screen shows a transient "Undo requested…" status so the player sees the request went out even when nothing reverts. OGS does **not** push a fresh snapshot on undo — it expects the client to step back locally — so `GameViewModel` keeps the full move list and, on `undo_accepted`, drops the taken-back move(s) (count from `parseUndoMoveCount`: the payload's `undo_move_count`, else **1**; legacy payloads are a bare move number) and rebuilds the position via `replayMoves`. Undo is offered (`undoEnabled`) only for **your own most recent move** — while `Playing`, with a move on the board, when it's the opponent's turn (the last stone is yours) and no request is pending; the button shows disabled otherwise. A real `game/<id>/move` from either side (the bot replying instead of accepting) clears the pending request. (Reverting replays from an empty board, so a **handicap** game with pre-placed stones not in the move list would mis-rebuild — fine for the even bot games we create. The accept→revert path is correct-by-construction — unit-tested `parseUndoMoveCount` + `replayMoves` — but **unverified live** pending a bot that accepts.)
- **Passing & scoring:** two consecutive passes (`game/move` with `".."`) move the game to `phase:"stone removal"` — the scoring phase, **not** the end. The board freezes until a player accepts: send `game/removed_stones/accept` `{game_id, player_id, stones, strict_seki_mode}` where `stones` is the dead-stone set in OGS encoding (echo the set the server/opponent marks — sourced from gamedata `removed` and the opponent's `game/<id>/removed_stones_accepted`; the strings must match exactly or the game won't conclude). `game/removed_stones/reject` resumes play. Bots auto-accept; once both agree, `phase` → `finished` with the scored result. Skipping this leaves any game that ends by passing frozen forever.
- **Puzzles:** **REST-only, no auth required** for public puzzles (we still send the token like every call, for a uniform network surface). `GET /api/v1/puzzles/collections/?ordering=<field>&page_size=N` = paginated browse list, **server-sorted** by `ordering` (`rating`, `puzzle_count`, `view_count`, `solved_count`, `created`, `min_rank`, `name`; prefix `-` for descending). Each entry: `{id, name, owner:{username}, puzzle_count, min/max_rank, rating, rating_count, view_count, solved_count, created, starting_puzzle:{id, width, height, initial_state}}`. (A `search=` param is **ignored** by this endpoint — don't rely on it.) `GET /api/v1/puzzles/{id}` = a single puzzle; `GET /api/v1/puzzles/{id}/collection_summary` = the **ordered `[{id,name}]`** of the whole collection (used for prev/next). Non-obvious bits, all handled in `net/Puzzles.kt`: **`initial_state`** is two **packed coordinate strings** (`{"black":"cacb","white":"dadb"}` — consecutive 2-char OGS coords, with `".."` pass placeholders; `decodePuzzleStones` chunks + decodes, skipping passes). **`initial_player`** names the colour the *human* plays (the opponent plays the other and its replies come from the tree). The **`move_tree`** is `{x,y,branches,correct_answer?,wrong_answer?}`: root is a no-move sentinel `x:-1,y:-1`; levels **alternate player → opponent → player…**; and OGS serializes an **empty `branches` as a string (`""`/whitespace), not `[]`** (so `parseMoveNode` only descends a real `JsonArray`, treating anything else as no children). **Crucially, `wrong_answer`/`correct_answer` often sit on the *opponent's refuting reply* a level deeper, not on the player's move** — so `puzzleStep(node, x, y)` evaluates the opponent's auto-reply too: no matching branch → `OffTree` (reject/flash); else it returns a `Play(playerMove, opponentMove?, outcome, next)` where `outcome` is `FAILED` if the player's move *or its refutation* is `wrong`, `SOLVED` if `correct` (or a dead-end line), else `CONTINUE` (advance to the reply node). The same deeper-flag fact is why `puzzleHints` is a minimax, not a non-wrong-branch filter. The ViewModel applies the moves with `BoardState.place` so captures render (life-and-death puzzles take stones).
- **GoTV:** `GET /api/v1/gotv/streams` returns a **bare JSON array** of *external* live Go streams (`{stream_id, channel, username, title, viewer_count, language, source, thumbnail_url, profile_image_url}`) — **not OGS games**. `source` is `"twitch"`/`"youtube"`; the watch URL is `twitch.tv/<channel>` or `youtube.com/watch?v=<channel>` (`streamUrl`). Privacy: open in the browser (no embed), and **don't fetch the `*_image_url`s** (Twitch CDN). Public endpoint (works without auth; we send the token anyway for uniformity).
- **Auth:** OAuth2 PKCE; the public `CLIENT_ID` in `Ogs.kt` is not a secret (PKCE, no client secret). `user_jwt` from `/api/v1/ui/config` authenticates the socket.

## Coding style

Follow `kotlin.code.style=official`. Mirror the existing files — they are consistent:

- **4-space indent; trailing commas** on multi-line argument/parameter lists.
- **Imports:** wildcard only for Compose (`androidx.compose.foundation.layout.*`, `material3.*`, `runtime.*`). Everything else (esp. `kotlinx.serialization.json` members) is imported explicitly. No wildcards outside Compose.
- **State modeling:** prefer `sealed interface` + `data object`/`data class` for UI/session state over nullable flags.
- **Comments explain *why*, not *what*** — especially the rationale behind privacy choices and OGS protocol quirks. Keep that habit; it's the house style.
- **Naming:** `*Screen` = Compose UI (body-only where a parent scaffold owns the top bar), `*ViewModel` = state holder. Private composables for a screen live in the same file.
- Compose: hoist state, pass `onX` lambdas down; ViewModels never reference Compose or Android UI types.

## Documentation and Testing Discipline

Code, its documentation, and its tests move together in the **same change** — never one without the others. This is what keeps the project from silently regressing or growing bugs outside the reach of the suite. Hold the line on all of it:

- **Extract logic so it can be tested.** Any non-trivial rule (game rules, parsing, encoding, request/response shaping) goes in a **pure top-level function** with no `Context`/`Android`/OkHttp/Compose deps — the `buildChallengeBody`, `parseActiveBots`, `replayMoves`, `parse*` pattern. If you find yourself unable to test something because it needs a socket or a `Context`, that's the signal to extract the pure core, not to skip the test. ViewModels and I/O classes should *call* these functions, not embed the logic inline.
- **Adding code → add tests.** New pure logic ships with unit tests covering the happy path **and** the edges that matter (boundaries, defaults/missing fields, the illegal/empty cases). No PR adds untested rule logic.
- **Fixing a bug → add the regression test first.** Reproduce it as a failing test, then fix it, so the bug can't come back. (e.g. a capture-count or ko miscalc gets a `replayMoves` case.)
- **Editing behavior → update the tests and the prose.** If you change what a function does, update its tests in the same commit; if you change a protocol/quirk/flow, update the relevant comment **and** the matching CLAUDE.md section (OGS cheat-sheet, Gotchas, etc.). Stale docs are worse than none.
- **Deleting code → delete its tests and docs.** Remove the now-dead tests, KDoc, and any CLAUDE.md / comment references in the same change. Don't leave orphans.
- **Keep the coverage list current.** When you add or remove a test file, update the bullet list under [Testing](#testing). Treat it as the index of what's actually guarded.
- **Document the *why*, not the *what*** (house style above) — especially privacy rationale and OGS protocol quirks. A KDoc on every public function/`data class`; a one-line `//` on any non-obvious branch.
- **What unit tests can't reach** (Compose UI, colors, real socket/REST timing), cover with the manual smoke flow below and say so in the PR — don't pretend a coverage gap doesn't exist.

## Non-negotiable constraints

These are the reason the app exists — verify any change preserves them:

- **`INTERNET` is the only permission.** No location, contacts, advertising ID, analytics, crash reporting, or third-party SDKs of that kind.
- **No analytics/telemetry**, including the *optional* OGS socket messages (`net/connects`, `net/timeout`, route latency, etc.). The `authenticate` payload stays minimal (jwt + client name; no device id / user agent).
- **Tokens encrypted at rest** (`TokenStore`); `allowBackup=false`, custom `data_extraction_rules`.
- **OAuth via Custom Tab (AppAuth), never an embedded WebView** — no token leakage.
- **Haptics use the no-permission path only** (`view.performHapticFeedback(...)` / `LocalHapticFeedback`). Do not add `VIBRATE` or use `Vibrator`/`VibrationEffect` — that would be a second permission. App info should always show zero runtime permissions.
- When adding a network call, ask: does this send more than strictly necessary? If unsure, send less.

## Gotchas

- Material3 dropdowns use the current `menuAnchor(MenuAnchorType.PrimaryNotEditable)` overload (the no-arg `menuAnchor()` is deprecated) — the anchor fields are read-only, hence `PrimaryNotEditable`.
- `Stone` has three values (`EMPTY`, `BLACK`, `WHITE`); `when` over a `Stone?` color needs to handle `EMPTY`/`else`, not just BLACK/WHITE/null.
- The socket's `events` flow has `replay = 0`; subscribe *before* connecting if you need a one-shot broadcast like `active-bots`.
